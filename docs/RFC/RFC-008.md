# RFC-008 - Storage Model and Derived Views Plan

**Status**: Draft for review  
**Owners**: Data Engineer, Backend Lead  
**Reviewers**: Analytics Lead, DevOps, Privacy Officer

## Context and Goals
- Sketch the relational layout for the ingest MVP.
- Define the derived views and tables that power analytics endpoints.
- Capture refresh cadence, zero-fill strategy, and primary access patterns.
- Pull the minimum raw storage slice into Phase 3 so ingest can persist accepted events immediately.

## Raw Event Storage (`events_raw`)
- Phase 3 now includes the initial Prisma schema and migration for `events_raw`.
- Append-only table on day one; daily partitioning remains a follow-on optimization once write patterns are confirmed.
- Columns:
  - `event_id`
  - `received_at`
  - `occurred_at`
  - `api_key_id`
  - `game_id`
  - `game_version`
  - `build_id`
  - `session_id`
  - `player_id_hash`
  - `event_name`
  - `schema_version`
  - `platform`
  - `locale`
  - `consent_analytics`
  - `props_jsonb`
  - `ingest_source`
  - `inserted_at`
- Primary key: `event_id` (UUID).
- Ingest source is stored as `godot_sdk` for MVP batches.
- Indexes:
  - time-range index on `received_at`
  - B-tree on `(game_id, event_name, occurred_at)` for aggregations
  - B-tree on `(player_id_hash, occurred_at)` for retention lookups
  - partial index on `event_name IN ('session_start', 'session_end')` to accelerate session derivations
- Retention: keep raw events for 90 days (per RFC-004), then prune via scheduled jobs.

## Derived Views and Tables

### `mv_sessions_daily`
- Materialized view refreshed every 5 minutes (concurrently).
- Aggregates session events into daily counts, average duration, and active players.
- Keys: `metric_date`, `game_id`, `consent_bucket`.
- Includes `suppressed` when counts are below 10.
- Zero-fill via join with `dim_dates` to ensure chart continuity.

### `mv_character_popularity`
- Materialized view refreshed every 5 minutes.
- Summarizes `character_selected` events by `game_id`, `metric_date`, `character_id`.
- Stores `pick_count`, `pick_ratio`, and suppression state.
- Index `(game_id, metric_date DESC)` to support rolling 7-day queries.

### `mv_retention_cohorts`
- Derived table populated nightly around `02:00 UTC`.
- Calculates cohorts from `session_start` events: `cohort_date`, `game_id`, `cohort_size`, `d1_retained`, `d7_retained`, suppression flags.
- Index `(game_id, cohort_date)` for dashboard sort and filter patterns.
- Allows manual refresh triggers for demos and recovery workflows.

### Supporting Structures
- `dim_dates` table for zero-fill.
- Future dimension tables such as `dim_characters` stay out of MVP scope.

## Refresh and Zero-Fill Strategy
- `mv_sessions_daily` and `mv_character_popularity` refresh every 5 minutes.
- `mv_retention_cohorts` refreshes nightly, with a manual rerun path available.
- Refresh jobs log duration and row counts to feed the RFC-005 observability metrics.
- Zero-fill uses generated date ranges joined against `dim_dates`; suppressed rows remain structurally present so APIs can return placeholders.
- When more than 50% of points in a requested window are suppressed, the API may signal an empty state to dashboard consumers.

## Access Patterns
- `/metrics/sessions/daily` reads from `mv_sessions_daily` filtered by `game_id` and date range.
- `/metrics/characters/popularity` reads from `mv_character_popularity` over the requested rolling window.
- `/metrics/retention/cohorts` reads from `mv_retention_cohorts` ordered by `cohort_date DESC`.
- Admin and debug queries may hit `events_raw` through indexed filters on `(game_id, event_name, occurred_at)`.

## Open Questions
1. Should we adopt TimescaleDB chunks instead of vanilla Postgres plus explicit partitioning later?
2. Do we need a `dim_players` table for retention analysis beyond hashed IDs?
3. How will we snapshot data before pruning old raw-event partitions?

## Decision
Adopt the above storage layout, pulled-forward raw ingest slice, refresh cadence, and access patterns for the MVP. The raw table is now implementation-ready; derived views and warehouse refresh logic remain follow-on work.
