# RFC-008 - Storage Model and Derived Views Plan

**Status**: Draft for review  
**Owners**: Data Engineer, Backend Lead  
**Reviewers**: Analytics Lead, DevOps, Privacy Officer

## Context and Goals
- Sketch the relational layout for the ingest MVP.
- Define the derived views and tables that power analytics endpoints.
- Capture refresh cadence, zero-fill strategy, and primary access patterns.
- Pull the minimum raw storage slice into Phase 3 so ingest can persist accepted events immediately.

## Raw Event Storage (`events_raw`)
- Phase 3 now includes the initial Prisma schema and migration for `events_raw`.
- Append-only table on day one; daily partitioning remains a follow-on optimization once write patterns are confirmed.
- Columns:
  - `event_id`
  - `received_at`
  - `occurred_at`
  - `api_key_id`
  - `game_id`
  - `game_version`
  - `build_id`
  - `session_id`
  - `player_id_hash`
  - `event_name`
  - `schema_version`
  - `platform`
  - `locale`
  - `consent_analytics`
  - `props_jsonb`
  - `ingest_source`
  - `inserted_at`
- Primary key: `event_id` (UUID).
- Ingest source is stored as `godot_sdk` for MVP batches.
- Indexes:
  - time-range index on `received_at`
  - B-tree on `(game_id, event_name, occurred_at)` for aggregations
  - B-tree on `(player_id_hash, occurred_at)` for retention lookups
  - partial index on `event_name IN ('session_start', 'session_end')` to accelerate session derivations
- Retention: keep raw events for 90 days, then prune via scheduled jobs.

## Derived Views and Tables

### `mv_sessions_daily`
- Materialized view refreshed through the warehouse worker `refresh:rolling` flow.
- Aggregates consented `session_end` events into daily counts, average duration, and active players.
- Keys: `metric_date`, `game_id`.
- Stores `session_count`, `active_players`, `avg_session_length_s`, `suppressed`, and `last_refreshed_at`.
- Zero-fill via join with `dim_dates` to ensure consumer continuity.

### `mv_character_popularity`
- Materialized view refreshed through the warehouse worker `refresh:rolling` flow.
- Summarizes consented `character_selected` events by `game_id`, `metric_date`, `character_id`.
- Stores `pick_count`, `pick_ratio`, `suppressed`, and `last_refreshed_at`.
- Remains sparse in storage; zero-fill is not applied here.

### `retention_cohorts`
- Derived table rebuilt through the warehouse worker `refresh:retention` flow.
- Calculates cohorts from consented `session_start` events using the first session day per `(game_id, player_id_hash)` as the anchor.
- Stores `cohort_date`, `game_id`, `cohort_size`, `d1_retained`, `d7_retained`, `d1_retention_pct`, `d7_retention_pct`, suppression flags, and `last_refreshed_at`.
- Uses `(cohort_date, game_id)` as the composite primary key.

### `mv_metrics_summary_current`
- Materialized view refreshed through the warehouse worker `refresh:rolling` flow.
- Stores the current KPI summary needed by `GET /metrics/summary`.
- Tracks `active_players_24h`, `matches_today`, `avg_session_length_s_24h`, suppression flags, and `last_refreshed_at`.
- Reads consented `session_end` and `match_end` events only.

### Supporting Structures
- `dim_dates` table for zero-fill.
- Future dimension tables such as `dim_characters` stay out of MVP scope.

## Refresh and Zero-Fill Strategy
- `apps/warehouse-worker` owns the local/manual refresh flow.
- `refresh:rolling` refreshes `mv_sessions_daily`, `mv_character_popularity`, and `mv_metrics_summary_current`.
- `refresh:retention` truncates and rebuilds `retention_cohorts`.
- `refresh:all` runs both flows for local rebuilds and demos.
- Zero-fill uses generated date ranges joined against `dim_dates` for daily sessions only; suppressed rows remain structurally present so APIs can return placeholders.
- When more than 50% of points in a requested window are suppressed, the API may signal an empty state to consumer UIs.

## Access Patterns
- `/metrics/sessions/daily` reads from `mv_sessions_daily` filtered by `game_id` and date range.
- `/metrics/characters/popularity` reads from `mv_character_popularity` over the requested rolling window.
- `/metrics/retention/cohorts` reads from `retention_cohorts` ordered by `cohort_date DESC`.
- `/metrics/summary` reads from `mv_metrics_summary_current`.
- Admin and debug queries may hit `events_raw` through indexed filters on `(game_id, event_name, occurred_at)`.

## Open Questions
1. Should we adopt TimescaleDB chunks instead of vanilla Postgres plus explicit partitioning later?
2. Do we need a `dim_players` table for retention analysis beyond hashed IDs?
3. How will we snapshot data before pruning old raw-event partitions?

## Decision
Adopt the above storage layout, pulled-forward raw ingest slice, refresh cadence, and access patterns for the MVP. The raw table, derived warehouse structures, and local refresh workflow are now implemented; analytics read APIs remain the next follow-on work.
